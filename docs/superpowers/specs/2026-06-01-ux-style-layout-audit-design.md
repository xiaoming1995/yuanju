# 全站 UX / 样式 / 布局优化审计设计

> **Status:** Draft · 用户已确认方向
> **Date:** 2026-06-01
> **Scope:** 前端用户端 + 管理后台的体验、样式一致性、布局合理性
> **Reference mockup:** `.superpowers/mockups/yuanju-high-fidelity-ux-mockup.html`

## 1. 背景

本次审计聚焦三个问题：

1. 用户能否在关键页面快速找到结论和下一步。
2. 页面之间的视觉语言、组件状态、按钮层级是否一致。
3. 移动端和后台密集信息布局是否稳定、可读、可操作。

项目当前已经完成 MVP 功能，但前端经历多轮功能叠加后，出现了典型的增长期问题：核心页面承载过多细节、局部样式各自生长、管理后台操作反馈不统一。

当前代码层面的信号：

| 信号 | 现状 |
|---|---|
| `ResultPage.tsx` | 1332 行，业务聚合与展示细节混在同一页面 |
| `ResultPage.css` | 2173 行，结果页样式已经成为局部设计系统 |
| `frontend/src/index.css` | 339 行，已有基础变量但覆盖面不足 |
| `style={{ ... }}` | 约 759 处，组件视觉不可复用 |
| 硬编码颜色 / `rgba()` / inline style 综合命中 | 约 1573 处 |
| `alert()` / `confirm()` | 约 36 处，后台和表单反馈不统一 |

这些问题不是单页美化能解决的，应按「用户流程优先 + 设计系统收敛」一起处理。

## 2. 优化原则

### 2.1 先流程，后视觉

核心页面先回答用户最关心的问题：

- 结果页：先给结论，再给依据，最后给行动入口。
- 合盘入口：先确认咨询问题和关系阶段，再填写双方出生资料。
- 后台：先完成筛选、定位、处理，不把用户困在长表格和弹窗里。

视觉优化必须服务这个顺序，不能只换颜色、加卡片或堆动画。

### 2.2 统一组件，而不是统一截图

本次不追求每页长得完全一样，而是统一这些基础部件：

- 页面容器：`PageShell`
- 区块容器：`SectionPanel`
- 主次按钮：`Button`
- 分段切换：`SegmentedTabs`
- 状态标识：`StatusBadge`
- 数据表格：`DataTable`
- 空 / 加载 / 错误状态：`EmptyState`、`Skeleton`、`InlineError`
- 危险操作确认：`ConfirmDialog`
- 操作反馈：`Toast`

页面可以有领域差异，但按钮、间距、圆角、标题层级、状态反馈不能各自定义。

### 2.3 深色金主题保留，但减少噪音

保留当前「深色 + 金色」的品牌方向。需要收敛的是使用方式：

- 金色只用于品牌、主行动、关键结论、当前选中态。
- 普通边框、说明文字、后台状态不滥用金色。
- 背景层级收敛到 3-4 层，避免多层半透明灰叠加。
- 五行色只用于命理语义，不作为普通 UI 装饰色。

### 2.4 移动端优先，桌面增强

用户端关键路径优先保证 375-430px 宽度下可读可点。桌面端用于增强信息密度，不把移动端强行缩成桌面表格。

后台则相反：桌面优先，移动端保证可用，不追求完整密集操作。

## 3. 审计结论与优先级

### P0 · 必须优先优化

| 模块 | 问题 | 目标态 |
|---|---|---|
| 结果页 `ResultPage` | 信息过长，用户要滚动寻找重点；主行动和专业细节权重接近 | 首屏只保留「命盘摘要、核心结论、主行动」；专业细节下沉为分段导航 |
| 历史 / 过往事件入口 | 结果页、历史页、过往事件之间的行动关系不够稳定 | 统一为「生成解读」「查看过往事件」「导出」三类行动 |
| 管理后台反馈 | 多处 `alert()` / `confirm()`，危险操作和成功反馈割裂 | 使用统一 `ConfirmDialog` + `Toast`，危险操作可撤销或二次确认 |
| 全局样式收敛 | inline style 和硬编码颜色过多 | 新增共享 UI primitives，逐批替换高频样式 |

### P1 · 第二批优化

| 模块 | 问题 | 目标态 |
|---|---|---|
| 合盘入口 `CompatibilityPage` | 表单像资料录入，不像咨询流程 | 改为「咨询问题 → 关系阶段 → 双方资料」的三段式入口 |
| 合盘结果 `CompatibilityResultPage` | 已有结构优化痕迹，但仍需和结果页共享视觉规则 | 对齐「结论优先、证据可展开、行动固定」的通用模式 |
| 后台列表页 | 筛选、状态、操作按钮样式不一致 | 统一 Toolbar + DataTable + StatusBadge + RowActions |
| Auth / Profile | 表单和账户页样式与主业务页割裂 | 使用同一表单布局、错误提示、按钮层级 |

### P2 · 第三批优化

| 模块 | 问题 | 目标态 |
|---|---|---|
| HomePage | 营销感与实际工具入口之间还可更直接 | 首屏明确「开始排盘 / 看历史 / 合盘」三个入口 |
| PDF / 分享卡 | 需要跟随页面结论层级做一致性校准 | 页面、分享卡、PDF 的结论和 CTA 文案保持一致 |
| 细节动效 | 当前无需新增复杂动效 | 只保留加载、展开、反馈的轻量过渡 |

## 4. 目标页面结构

### 4.1 结果页

目标：用户一进页面就知道「这张盘怎么看、下一步做什么」。

首屏结构：

1. 命盘身份摘要：年月日时柱、性别、出生时间。
2. 核心结论：一句用户能理解的判断。
3. 主行动：生成 AI 解读。
4. 次行动：查看过往事件、导出 PDF。
5. 分段导航：总览、命盘、用神、大运、AI 解读。

细节规则：

- 专业术语不在首屏堆满，放到分段导航内。
- 移动端底部固定主 CTA，避免用户滚到底找按钮。
- 标题层级固定：页面标题、区块标题、卡片标题、说明文字四级。
- 结果页不再承担局部设计系统，样式沉到共享组件。

### 4.2 合盘入口

目标：从「填表」改成「发起一次关系咨询」。

推荐三段：

1. 你想问什么：关系定位、关注主题、是否需要 AI 深度解读。
2. 现在处在什么阶段：认识中、暧昧、恋爱、婚姻、合作等。
3. 双方出生资料：两张命盘资料卡，支持从历史命盘选择。

细节规则：

- 出生资料仍完整保留，但不是第一视觉重点。
- 错误提示就地展示，不弹窗。
- 提交前显示「将生成什么结果」的轻量确认。

### 4.3 合盘结果

目标：和结果页保持同一阅读节奏。

结构：

1. 双方是谁。
2. 是否合。
3. 为什么这么判断。
4. 怎么相处。
5. 专业依据。

细节规则：

- 结论和分数放在首屏。
- 证据默认可读，但允许折叠。
- AI 深度分析不和量化评分抢权重。

### 4.4 管理后台

目标：后台优先保证效率、可扫描、少打断。

统一结构：

1. 左侧导航。
2. 页面标题 + 关键指标。
3. Toolbar：搜索、筛选、主操作。
4. DataTable：状态、时间、操作者、操作按钮。
5. Toast / ConfirmDialog 统一反馈。

细节规则：

- 危险操作必须使用统一确认弹窗。
- 表格按钮密度降低，次要操作进入更多菜单。
- 状态标签统一颜色和文案，不每页自定义。
- 后台不用大面积品牌装饰，保持工具感。

## 5. 设计系统收敛范围

### 5.1 Token

需要统一的变量：

- 颜色：背景、表面、边框、正文、辅助文字、金色强调、危险、成功、警告。
- 字号：页面标题、区块标题、卡片标题、正文、说明、小标签。
- 间距：页面边距、区块间距、卡片内边距、表单行距。
- 圆角：按钮、输入框、卡片、弹窗。
- 阴影：仅弹窗和浮层使用，不给普通内容卡滥用阴影。

### 5.2 组件

第一批组件建议：

| 组件 | 作用 |
|---|---|
| `PageShell` | 页面宽度、顶部间距、移动端安全区 |
| `SectionPanel` | 标题、说明、内容区统一 |
| `Button` | primary / secondary / ghost / danger |
| `SegmentedTabs` | 结果页、合盘页分段导航 |
| `StatusBadge` | 后台状态、AI 调用状态、命盘标签 |
| `DataTable` | 后台表格基准 |
| `ConfirmDialog` | 替换 `window.confirm()` |
| `ToastProvider` | 替换 `alert()` |
| `FormField` | label、hint、error 统一 |
| `EmptyState` | 空数据和错误恢复入口 |

## 6. 验收标准

### 用户体验

- 结果页首屏能看到核心结论和主行动。
- 合盘入口不再像单纯资料表单，用户能先表达咨询问题。
- 后台危险操作不再触发浏览器原生弹窗。
- 空、加载、错误状态不再留白或跳出浏览器弹窗。

### 样式一致性

- 新增页面改动不得继续引入大段 inline style。
- 新增颜色必须先进入 token，不能直接硬编码。
- 主按钮、次按钮、危险按钮在用户端和后台保持一致。
- 状态标签文案和颜色有统一映射。

### 布局

- 375px 移动端无横向溢出。
- 结果页和合盘结果页的主 CTA 在移动端始终可达。
- 后台表格在 1280px 桌面下不挤压核心字段。
- 文本优先换行，不靠缩到 10px 以下解决拥挤。

### 工程

- `npm run lint` 通过。
- `npm run build` 通过。
- 为关键共享组件补充源码级测试或渲染测试。
- 主要页面至少保留桌面和移动端截图作为人工验收依据。

## 7. 非目标

- 不重写后端接口。
- 不引入 Ant Design、MUI、TailwindCSS 等 UI 框架。
- 不一次性重构所有页面 TSX。
- 不做品牌大改版。
- 不改八字算法、AI prompt、计费或权限逻辑。

## 8. 推荐实施路线

推荐按 5 批推进：

1. 建立 UI token 和基础组件。
2. 改结果页首屏和行动结构。
3. 改合盘入口和合盘结果一致性。
4. 改后台 Shell、表格、反馈。
5. 收尾 Auth / Profile / Home / PDF 分享一致性。

每批都应能独立构建、独立验收，避免一次性大改造成回归难以定位。
