# 丧吊类神煞（丧门·披麻·三丘·五墓）实现设计文档

日期：2026-06-07
状态：已通过 brainstorming 评审，待用户复核

## 1. 目标

在现有神煞系统中新增「丧吊类」凶煞 **丧门、披麻、三丘、五墓** 四个，使其出现在：

- **个人命盘**（四柱神煞）
- **大运**
- **流年**（与大运共用同一计算函数，自动覆盖）

**合盘本轮不做**——合盘目前完全不展示任何神煞，给它开神煞通道是另一个独立功能，不在本次范围。

系统已有同族的 **吊客、墓门**，本次补齐丧吊一族。

## 2. 算法口径（对齐主流年支起例 + sohu 来源季节表）

四个神煞的基准与匹配方式**不完全一致**，这是忠于原始查法，不是疏漏：

| 神煞 | 基准 | 匹配粒度 | 规则 |
|---|---|---|---|
| 丧门 | 年支 | **地支** | 年支 +2 位 |
| 披麻 | 年支 | **地支** | 年支 −3 位 |
| 三丘 | 月支(季节) | **全干支** | 春→辛丑、夏→壬辰、秋→乙未、冬→丙戌 |
| 五墓 | 月支(具体月) | **全干支** | 寅卯→乙未、巳午→丙戌、申酉→辛丑、亥子→壬辰、季月(辰未戌丑)→戊辰 |

- 丧门/披麻：和已有「吊客」（年支−2，地支匹配）同一套路，只比地支。
- 三丘/五墓：用户明确选择「严格全干支匹配」（设计讨论中的 B 方案），和已有「魁罡/十恶大败」全柱匹配同档。

### 2.1 丧门完整映射（年支 → 命中地支）

子→寅、丑→卯、寅→辰、卯→巳、辰→午、巳→未、午→申、未→酉、申→戌、酉→亥、戌→子、亥→丑

### 2.2 披麻完整映射（年支 → 命中地支）

子→酉、丑→戌、寅→亥、卯→子、辰→丑、巳→寅、午→卯、未→辰、申→巳、酉→午、戌→未、亥→申

### 2.3 三丘完整映射（月支 → 命中全干支）

按季节定（季节由月支落在哪一组判断）：

| 月支(季) | 三丘干支 |
|---|---|
| 寅卯辰（春）| 辛丑 |
| 巳午未（夏）| 壬辰 |
| 申酉戌（秋）| 乙未 |
| 亥子丑（冬）| 丙戌 |

### 2.4 五墓完整映射（月支 → 命中全干支）

按具体月定，四季月（辰未戌丑月）特殊：

| 月支 | 五墓干支 |
|---|---|
| 寅、卯（正二月）| 乙未 |
| 巳、午（四五月）| 丙戌 |
| 申、酉（七八月）| 辛丑 |
| 亥、子（十·十一月）| 壬辰 |
| 辰、未、戌、丑（三六九腊·季月）| 戊辰 |

> 注：三丘与五墓在春夏秋冬上互为对冲（春丑/未、夏辰/戌、秋未/丑、冬戌/辰），但五墓在四季月有独立的「戊辰」规则，因此不能简单用「三丘取冲」实现，必须按上表显式映射。

### 2.5 匹配范围说明

- **四柱**：扫描年/月/日/时四柱。丧门/披麻比每柱地支；三丘/五墓比每柱全干支（天干+地支）。
- **大运**：比大运柱（运干+运支）。丧门/披麻比运支；三丘/五墓比运柱全干支。
- **流年**：复用大运函数（`event_signals.go` 传流年代替大运），自动覆盖，无需额外改动。
- 自我命中问题：三丘/五墓的目标干支永不等于其判定月支所属柱，丧门/披麻目标地支永不等于年支自身，故四柱扫描不会出现基准柱自标，无需排除逻辑。

## 3. 数据流（已查证）

神煞只有两个计算入口，所有消费方都从这两处取数：

| 入口函数 | 文件 | 写入 | 覆盖范围 |
|---|---|---|---|
| `GetPillarsShenSha(yg,yz,mg,mz,dg,dz,hg,hz)` | `pkg/bazi/shensha.go` | `BaziResult.{Year,Month,Day,Hour}ShenSha`（`engine.go:383`）| 个人命盘报告 |
| `GetDayunShenSha(yg,yz,mz,dg,dz,运干,运支)` | `pkg/bazi/shensha_dayun.go` | `DayunItem.ShenSha`（`engine.go:361`）；流年复用（`event_signals.go:1310`）| 大运 + 流年 |

`GetPillarsShenSha` 已接收 `mz`（月支），`GetDayunShenSha` 已接收 `monthZhi`，**三丘/五墓所需的月支基准两个函数都现成有**，无需改函数签名。

前端展示：

- 极性来自 `ShenShaPolarity` map（`shensha.go`），`xiong` → 红色徽章。
- 释义来自 `shensha_annotations` 表（`GET /api/shensha/annotations`），按 name 匹配。现有 40 条含吊客/墓门，**不含本次 4 个**，需新增 seed，否则前端只显示名字无解释。

## 5. 空值与边界

- 月支/年支为空或非法 → 对应 map 查不到，跳过该神煞，不 panic（沿用现有 `map[...]; ok` 与 `addIf` 写法）。
- 三丘/五墓全干支匹配：地支相同但天干不同（如盘中有「己丑」而三丘目标为「辛丑」）→ **不命中**（B 方案的核心特性，需有测试反例守住）。
- 旧报告/已生成数据不回填：神煞为生成时计算，历史已生成的报告不受影响（符合现有行为，本次不做数据回填）。

## 6. 验证标准

1. `go test ./pkg/bazi/...` 全绿（含新增样例）。
2. `go build ./...` 通过。
3. 个人命盘报告中，构造一个已知命中盘 → 对应柱出现「丧门/披麻/三丘/五墓」，且前端为红色凶煞徽章、有释义（migration 应用后）。
4. 大运/流年中，命中运柱出现对应神煞。
5. 全干支严格性：地支对天干错的盘 → 三丘/五墓**不**出现。

## 7. 明确不做（YAGNI）

- 不动合盘（`compatibility_service.go` 本轮不加任何神煞通道）。
- 不回填历史已生成报告。
- 不新增白虎/病符等其他太岁神煞（本次只做用户确认的四个）。
- 不改神煞计算函数签名、不重构现有神煞计算逻辑。
- 不为三丘/五墓做「地支匹配」开关（已定 B 方案，单一口径）。
