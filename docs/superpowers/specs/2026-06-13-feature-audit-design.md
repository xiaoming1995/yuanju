# 全面功能审计设计（方案一：分维度并行扫描）

> Status: Approved
> Date: 2026-06-13
> Scope: 全项目（frontend + backend）功能/体验、性能、代码质量三维审计
> 产出物: `docs/superpowers/specs/2026-06-13-feature-audit.md`（带勾选框的优先级报告）

## 1. 背景

CSS token 对齐（2026-06-12）完成后，用户要求全面审计项目还有哪些功能可以优化。本设计定义审计的范围、方法、汇总规则与产出物格式。审计只产出报告，不改任何代码。

## 2. 三路扫描

| 路 | 方法 | 关注点 |
|---|---|---|
| ①功能/体验 | 沿用户旅程走读路由和页面代码：起盘（生日/八字反查）→ 结果页 → 深度报告 → 合婚 → 历史 → 分享/导出 → 个人资料/品牌设置 → admin | 流程断点、半成品功能、错误处理缺失（加载/失败/空态）、移动端适配缺口、明显交互别扭处 |
| ②性能 | 跑 `npm run build` 拿真实产物数据 + 代码静态检查 | 包体积与分包（chunk 是否过大、有无路由级懒加载）、大依赖、明显的重复请求/无缓存、后端代码里的明显浪费 |
| ③代码质量 | 静态扫描 frontend/src + backend | 超长文件（对照 ENGINEERING.md 行数约束）、第 3 次复制未抽离的重复代码、缺测试的核心模块、TODO/FIXME 清点、CSS 建议清单未决策项回链 |

## 3. 汇总规则

- 每条 finding 必须带 `文件:行号` 或构建输出数据作证据，无证据不进报告。
- 优先级：**P0** 影响用户正常使用的缺陷/断点；**P1** 明显改善体验或省维护成本；**P2** 锦上添花。
- 每条标预估工作量：S（<半天）/ M（半天–2天）/ L（>2天）。
- 跨路重复的合并；纯主观偏好不进报告。

## 4. 产出物格式

按 P0/P1/P2 分节，每条 `- [ ]` 勾选框 + 证据 + 建议做法 + 工作量，格式对齐 `2026-06-12-css-style-suggestions.md`。用户勾选后另起计划执行。

## 5. 边界

- 不改任何代码、不启动服务（构建除外）。
- admin 后台按内部工具标准看，不按 C 端体验要求挑刺。
- 命理算法正确性不在范围。
