# 合盘咨询型报告升级设计

## 背景

当前合盘功能已经完成基础闭环：用户可输入双方生辰，系统生成四维评分、缘分时长评估、结构化证据、AI 合盘解读和合盘历史。现有产品更接近“合盘测试结果页”，能回答“合不合、分数多少、有哪些依据”，但还不足以形成高信任的咨询型体验。

本次优化目标是把合盘升级为“综合型咨询报告”：默认回答婚恋决策问题，兼顾关系经营建议，并为命理爱好者和专业用户保留可验证的专业依据展开层。

## 目标

- 让普通用户在首屏看懂关系判断、投入建议、主要风险和下一步行动。
- 让懂命理的用户能展开验证双盘摘要、夫妻宫、十神互动、合冲刑害、五行互补等依据。
- 让每条关键判断都能回链到结构化证据，避免 AI 自由发挥成为唯一可信来源。
- 在 AI 报告未生成或生成失败时，页面仍能展示完整的结构化诊断骨架。
- 所有关系判断使用倾向、概率、条件和风险语言，避免具体分手日期、必然结婚、必然分手等绝对断语。

## 非目标

- 不预测具体结婚日期、分手日期或精确月份级结果。
- 不做多人对象横向比较。
- 不做关系复盘日记、长期追踪或关系 CRM。
- 不做命理师工作台、咨询师入驻或人工批注流程。
- 不做社交分享裂变或传播海报。

## 产品定位

第一阶段定位为“综合型咨询合盘报告”。

默认用户问题是：“这段关系值不值得继续投入，风险在哪里，怎么相处更好？”因此页面不能只给分数和命理术语，而应按咨询报告逻辑组织：

1. 关系诊断总览
2. 决策建议
3. 阶段风险预警
4. 关系经营策略
5. 专业依据展开
6. AI 完整解读

## 信息架构

### 1. 关系诊断总览

首屏给出一组稳定、结构化的诊断信息：

- `relationship_type`：关系类型，例如“高吸引高磨合型”“稳定经营型”“短期吸引强、长期承压型”。
- `verdict`：总体判断，例如“适合继续推进”“建议谨慎观察”“不宜过早重投入”。
- `top_findings`：最多 3 条关键发现，每条需要有对应证据引用。
- `summary`：一段白话短结论，强调倾向与条件。

### 2. 决策建议

决策建议回答用户是否值得继续投入：

- `recommendation`：`continue` / `observe` / `caution`。
- `confidence`：`high` / `medium` / `low`，表达系统判断把握度，不展示伪精确百分比。
- `conditions`：继续推进的前提条件，例如“先稳定沟通节奏”“现实安排需要对齐”。
- `do_next`：建议用户下一步做什么。
- `avoid`：短期内应避免的相处模式。

### 3. 阶段风险预警

基于已有 `duration_assessment` 扩展为更咨询化的阶段结构：

- `window`：`three_months` / `one_year` / `two_years_plus`。
- `risk_level`：`high` / `medium` / `low`。
- `main_risk`：该阶段最主要风险。
- `trigger`：风险通常在什么情境下被触发。
- `advice`：该阶段最重要的处理建议。
- `evidence_ids`：支撑该判断的证据列表。

### 4. 关系经营策略

把命理信号转化为可执行建议：

- `communication`：沟通方式建议。
- `conflict`：冲突处理建议。
- `reality`：现实安排、节奏、责任分工建议。
- `boundary`：边界感与投入节奏建议。

每条策略应尽量关联至少一条证据或阶段风险，不能只给泛泛鸡汤。

### 5. 证据链引用

新增关键判断到证据的映射层：

- `claim_id`：稳定标识。
- `claim`：用户可读判断。
- `evidence_ids`：对应的结构化证据。
- `reasoning`：一段短推理，说明证据如何支持判断。
- `caveat`：限制条件或反向因素，避免单证据过度推断。

这是专业可信的核心机制。前端可在关键结论旁展示“查看依据”，展开后显示证据卡和推理链。

### 6. 专业依据展开

保留折叠式专业层，服务命理爱好者和高阶用户：

- 双方四柱与日主摘要。
- 五行互补与共同短板。
- 夫妻宫互动。
- 双向配偶星/十神互动。
- 干支合冲刑害摘要。
- 关键神煞辅助信号。

首屏不铺满术语，专业信息通过展开层呈现。

## 分工边界

### 算法层

算法层负责可信事实和结构化判断：

- 生成合盘证据，并为每条证据提供稳定 `evidence_id`。
- 输出四维评分、阶段风险底分、关系类型标签。
- 输出 `claim_evidence_links` 的初始映射。
- 给出保守的决策建议底层分类。
- 明确不输出具体日期级断语或绝对命运结论。

### AI 咨询层

AI 负责把结构化证据组织成咨询语言：

- 输出固定 JSON，而不是自由散文。
- 将算法证据翻译成普通用户能理解的诊断、风险和建议。
- 生成关系经营策略和阶段说明。
- 不得新增没有证据支撑的命理判断。
- 对不确定项使用“倾向于”“更容易”“需要观察”等条件语言。

### 前端层

前端负责分层阅读体验：

- 首屏突出关系诊断、决策建议和关键发现。
- 阶段风险用时间窗口卡片表达，避免伪精确时间预测。
- 每条关键结论提供“查看依据”展开能力。
- 专业命盘与证据集中在专业展开层。
- AI 完整解读放在结构化咨询模块之后，作为深入阅读内容。

## 第一阶段范围

第一阶段交付以下能力：

1. 关系诊断总览。
2. 决策建议。
3. 阶段风险预警。
4. 关系经营策略。
5. 关键判断到证据的引用。
6. 专业依据展开层增强。

第一阶段可以在现有合盘数据模型上增量扩展，不要求重建整个合盘资源模型。

## 验收标准

- 用户不用懂命理，也能在首屏理解关系判断和建议。
- 每条关键建议都能展开找到对应命理证据。
- 懂命理用户能看到双方四柱、核心合冲刑害、十神互动和夫妻宫依据。
- AI 报告未生成时，页面仍展示结构化诊断、阶段风险和证据层。
- AI 输出不能包含未由证据支撑的新命理判断。
- 文案避免“必然”“注定”“一定分手”“一定结婚”等绝对措辞。
- 合盘历史仍与单人命盘历史隔离。

## 风险与约束

- 算法证据不足时，咨询建议容易显得空泛；应优先保证关键判断有证据链。
- 如果一次性展示过多专业术语，普通用户会降低阅读完成率；专业层应默认折叠或半折叠。
- AI JSON 输出需要容错和 fallback，避免模型偶发格式错误导致结果页空白。
- 现有历史记录可能缺少新增字段，应提供前端 fallback 或后端懒生成补齐。

## 后续方向

- 增加双向十神矩阵与更完整的专业推理链。
- 增加深度付费报告，扩展更多阶段建议和案例化解释。
- 增加关系复盘能力，但应作为独立变化处理。
- 增加对象对比能力，但不进入第一阶段。
