# 毛泽东思想指导下的工程实践

> "没有调查就没有发言权。" — 编码之前，先读懂代码库。

### 一、动手之前：调查再出手（实事求是）

**严格禁令**：不看现有代码就开写，等同于闭门造车，严惩。

- **必须先检索**：动手前用 `grep` / `find` 扫描以下关键目录，确认相关逻辑不存在重复实现
  - 后端新功能 → 先查 `backend/internal/handler/`、`backend/internal/service/`、`backend/internal/repository/`
  - 后端新算法 → 先查 `backend/pkg/bazi/`（engine.go / shishen.go / shensha.go）
  - 前端新组件 → 先查 `frontend/src/components/`、`frontend/src/lib/`
  - 前端新页面 → 先查 `frontend/src/pages/`，确认路由未重叠（`App.tsx`）
  - 数据库变更 → 先查 `backend/pkg/database/database.go`（DDL 集中在此）
- **必须先阅读**：读懂受影响文件的现有逻辑，再设计方案，不凭主观想象
- **必须先确认**：理解需求偏差率必须为 0%，有歧义先问清楚

### 二、动手之中：抓主要矛盾

**方法论**：接到任务 → 找核心问题 → 集中火力解决 → 次要问题依次处理

- **聚焦主要矛盾**：先解决关键路径上的问题，不要眉毛胡子一把抓
- **最小化改动范围**：不修改任务范围之外的任何代码，改动前评估影响范围
- **禁止随意引入新依赖**：引入新包前必须说明理由，并确认项目现有方案无法满足
- **禁止过度设计**：一个计数器不需要工厂模式 + 观察者模式，够用就好
- **禁止文件膨胀**：单文件超过 500 行未拆分，是对代码库的犯罪

### 三、动手之后：实践检验

**认识论**：写代码 → 跑起来 → 发现问题 → 改进 → 循环往复，螺旋上升

- 代码写了必须能跑，跑了才算完成，不追求一步到位的完美方案
- 改动后主动验证，不假设「应该没问题」
- 复用优先级高于新建：发现已有组件 / 工具函数，必须复用，不重复造轮子

### 四、三大铁律（不可逾越）

```
1. 理解需求偏差率 = 0%    有歧义先问，不要自行脑补
2. 不动无关代码           改 A 不破 B，影响范围最小化
3. 一切缴获要归公         发现的好方案及时记录，沉淀到项目规范
```

### 五、反面教材（禁止重蹈）

| 反模式 | 对应原则 |
|--------|----------|
| 没看现有代码就新建同功能模块 | 实事求是 |
| `// TODO: 优化性能` 留了三年没动 | 反对形式主义 |
| 顺手重构了任务范围外的代码 | 最小化改动 |
| 同时开三个分支、改了 A 破了 B | 抓主要矛盾 |
| 引入新依赖但项目现有方案已够用 | 禁止引入新依赖 |
| 代码覆盖率 100% 但全是无意义测试 | 反对虚荣指标 |
