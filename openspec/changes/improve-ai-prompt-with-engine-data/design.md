## Context

当前 `buildBaziPrompt()` 函数只传入四柱干支、五行统计、藏干、纳音，约要求 LLM 自行「在心中」完成日主强弱判断和用神推定。而引擎（`bazi.Calculate()`）已精确计算出十神、十二长生、神煞、旬空、初步用神等数据，这些数据完全未被利用于 Prompt 构建。

LLM 的排盘推算准确性不可控，且其输出结论可能与算法不一致（例如前端 UI 显示的十神与 AI 报告叙述不符）。此次改动收窄 LLM 职责边界：**引擎负责排盘，LLM 负责解读**。

## Goals / Non-Goals

**Goals:**
- 将引擎精算的十神、十二长生、神煞、旬空、完整大运（10步）注入 Prompt
- 将引擎初步用神/忌神作为「参考提示」传入，LLM 可基于更完整数据确认或微调
- CoT 步骤从「自行推算」改为「整合精算数据后综合判断」
- 报告新增「大运走势」章节，LLM 基于完整大运数据解读人生各阶段
- `max_tokens` 从 2000 上调至 3500，适配更长报告

**Non-Goals:**
- 不修改 `BaziResult` 数据结构（所需字段均已存在）
- 不修改前端展示逻辑
- 不修改数据库 Schema 或 API 接口
- 不修改其他 Provider（`callAI`、`callOpenAICompatible` 无需改动）

## Decisions

### 1. Prompt 数据注入结构

将引擎数据按语义区块注入，LLM 更易消化：

```
===命盘精算数据===
四柱 + 五行 + 藏干（现有内容保留）

===十神关系===
年干[X]：十神=[正官] | 月干[X]：十神=[偏财] | ...
年支[X]：主气十神=[伤官] | 月支[X]：主气十神=[...] | ...

===十二长生（日主星运）===
年柱对日主：[胎] | 月柱：[养] | 日柱：[长生] | 时柱：[沐浴]

===神煞===
年柱：[天乙贵人 驿马] | 月柱：[文昌] | ...

===大运序列===
（1）X岁～X岁：[甲子] 干十神=[正印] 支十神=[正官]
...（共10步）

===引擎初步推算===
引擎基于五行统计初步判断：用神=[木火]，忌神=[金水]（供参考，请综合十神数据确认或微调）
```

### 2. CoT 步骤重构

原「第一步：在心中完成专业分析」改为：

> 「第一步：综合以上精算数据（十神关系、日主星运、神煞特质），进行专业整合判断：
> 1. 结合月令十神和日主星运，综合评估日主得令/失令情况；
> 2. 参考引擎初步用神，结合十神全貌确认或微调用神/忌神；
> 3. 归纳2~3个与此人命局最相关的神煞特质。
> 以上步骤在心中完成，不要在报告中输出推理过程。」

### 3. 报告章节从四章扩为五章

新增第五章：**【大运走势】** —— 结合起运年龄和各大运干支十神，解读人生各阶段的整体运势节奏，重点分析当前大运和近1~2步大运对事业、感情的影响。

### 4. max_tokens 从 2000 → 3500

五章报告 + 更多精算数据 → 输出 token 需要更多空间。3500 在各主流 Provider（DeepSeek / Kimi / OpenAI）均支持且未触及上限。

### 5. 神煞格式处理

`ShenSha` 字段为 `[]string`（空时为 `[]`），注入时过滤空数组并展示为逗号分隔文本，若无神煞则显示「无」。

## Risks / Trade-offs

- **Prompt 变长导致 token 消耗增加**：大运10步 + 神煞 + 十神注入后 prompt token 约增加 400~600 个。对所有收费 Provider 有成本影响。→ 暂不限制，作为数据完整性的合理代价。
- **LLM 忽略参考用神**：LLM 可能不采纳引擎初步用神，给出不同结论——这是设计允许的（B方案：LLM 可微调）。
- **某些 Provider 对长 Prompt 敏感**：个别免费 API（如部分 Kimi 套餐）有 token 限制。→ 在 Admin 面板中可切换 Provider，问题可控。
- **大运流年数据未传入**：10步 × 10流年 = 100条数据量过大，设计上只传大运本身（甘支+十神），不含流年细节。
