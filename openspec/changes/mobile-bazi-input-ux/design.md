## Context

当前出生信息输入由 `BirthProfileForm` 承载性别、历法、日期和时辰，首页再追加早子时与出生省份。这个结构功能完整，但在移动端会把基础起盘和专业校准混在同一条表单路径上，用户需要在提交前理解“农历闰月、早子时、真太阳时修正”等概念。

本次设计以移动端为主约束，同时保持桌面端简洁。后端 `/api/bazi/calculate` 已支持 `calendar_type`、`is_leap_month`、`is_early_zishi`、`longitude`，因此不需要改变接口契约。

## Goals / Non-Goals

**Goals:**

- 让普通用户在手机上以最少认知成本完成起盘输入。
- 保留专业用户需要的出生地真太阳时修正和子时细分能力。
- 在提交前展示摘要，帮助用户确认历法、日期、时辰和校准方式。
- 复用现有 `BirthProfileForm`，让首页起盘与合盘输入在基础体验上保持一致。
- 不引入 UI 组件库，继续使用现有 CSS Variables 和本地样式体系。

**Non-Goals:**

- 不修改八字算法、命盘结果结构或 AI 报告生成逻辑。
- 不新增后端 API 或数据库字段。
- 不在本次支持“不知道出生时辰”的降级排盘模式。
- 不引入城市级定位、GPS 定位或外部地理编码服务。

## Decisions

### D1: 移动优先的基础输入分层

基础输入只展示性别、历法、出生日期和出生时辰。移动端采用单列为主的布局，年份单独一行，月日可在空间允许时并排，否则自然堆叠。桌面端可以继续使用更紧凑的多列布局。

替代方案是维持当前所有字段顺序并只调 CSS。这个方案改动小，但不能解决专业项抢占注意力的问题。

### D2: 高级校准默认折叠

出生省份、真太阳时说明和子时细分归入“高级校准”。默认按北京时间起盘；用户展开后才能调整出生地和子时细分。

这样保持普通用户的路径短，同时不会移除专业能力。子时细分仍可在选择子时后通过更明确的双选项出现。

### D3: 用确认摘要替代表单解释文本

表单底部展示一条摘要，例如：

`男命 · 公历 1998年6月12日 午时 · 按北京时间排盘`

当用户选择农历、闰月、出生省份或早子时时，摘要同步变化。摘要承担“确认输入”的职责，避免在界面里堆砌说明文字。

### D4: 早子时改为明确选项

当前 checkbox 文案容易让用户不确定“勾选代表什么”。新交互在子时时显示两个互斥选项：

- `23:00-23:59，按前一日`
- `00:00-00:59，按当日`

提交时仍映射为现有接口字段：前者传 `hour=23` 与 `is_early_zishi=true`，后者传 `hour=0` 与 `is_early_zishi=false`。

### D5: 通用组件扩展而不是页面内重复实现

`BirthProfileForm` 应扩展为可配置组件：基础字段、确认摘要和高级校准可由页面按场景启用。首页需要完整校准能力；合盘页至少应获得基础输入体验，是否接入高级校准可作为实现任务中的明确选择。

## Risks / Trade-offs

- **[隐藏高级项导致专业用户找不到]** -> 使用清晰的“高级校准”折叠标题，并在摘要中展示当前是否按北京时间或出生地修正。
- **[移动端 sticky 按钮遮挡内容]** -> 若使用 sticky 提交区，必须为表单底部预留安全间距，并验证小屏视口。
- **[组件抽象过度]** -> 只抽取真实复用的出生信息输入逻辑，避免为了首页和合盘差异引入复杂配置矩阵。
- **[农历闰月摘要遗漏]** -> 摘要生成必须覆盖 `calendarType` 和 `isLeapMonth`，避免用户看不到闰月状态。

## Migration Plan

1. 先升级 `BirthProfileForm` 的布局、摘要和子时表达。
2. 首页接入高级校准区域，保持现有请求字段映射不变。
3. 合盘页接入升级后的基础输入体验，确认不会破坏双人表单布局。
4. 使用桌面和移动视口手动验证首页起盘、农历日期、子时细分和合盘创建。
