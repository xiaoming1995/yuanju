## Context

当前 PDF 导出使用前端打印布局组件 [PrintLayout.tsx](/Users/liujiming/web/yuanju/frontend/src/components/PrintLayout.tsx) 渲染，再由结果页的打印 / PDF 导出入口复用该布局完成输出。命格数据 `ming_ge` 与 `ming_ge_desc` 已存在于结果页数据中，且屏幕端已经通过 badge 与弹窗展示，但打印布局没有接收这两个字段，因此最终命书无法直接说明“命主是什么格”。

这次变更只针对 PDF 内容编排做增强，不触碰后端算法、AI prompt 或结构化报告协议。现有 `analysis.logic` 已经围绕系统主格生成，因此可以作为“本局落点”的辅助来源，但不应成为新的强依赖。

## Goals / Non-Goals

**Goals:**
- 在 PDF 打印布局中新增独立的“命格解读”区块
- 区块展示命格名称与标准格义，并在可用时补充一句本局落点
- 让新内容位于“命局分析总览”之前，形成更顺的阅读顺序
- 保持实现为纯前端展示增强，不要求后端新增字段或新 AI 章节

**Non-Goals:**
- 不新增后端接口、数据库字段或 AI 结构化字段
- 不改动结果页现有命格 badge / 弹窗交互
- 不尝试做复杂 NLP 摘要或二次推理
- 不调整现有 PDF 导出机制（仍沿用打印布局 + 前端导出链路）

## Decisions

### D1. 命格解读只进入打印布局，不先扩展 AI 结构

本次需求聚焦“PDF 内容优化”，最直接的落点是 `PrintLayout`。相比新增 `analysis.mingge` 或新章节，这样可以直接复用已有 `ming_ge` / `ming_ge_desc`，避免把一个展示增强扩成完整的 AI 协议变更。

备选方案是新增正式 AI 字段，但那会同时涉及 prompt、解析、前端结构化渲染和回归验证，明显超出当前范围。

### D2. 命格解读区块放在“命理解读”页顶部，先于命局分析总览

打印阅读顺序调整为：

```text
命格解读
→ 命局分析总览
→ 性格 / 感情 / 事业 / 健康 / 大运
```

这样 PDF 会先回答“你是什么格”，再展开“这个格在你的盘里如何运作”。相比放在封面区，这个位置更接近解读正文，也更像命书内容而非标签信息。

### D3. 区块使用三段式结构，并对“本局落点”采用保守提取

区块内容固定为：
- 主格：`ming_ge`
- 格义：`ming_ge_desc`
- 本局落点：可选，来源于 `structured.analysis.logic`

“本局落点”不显示整段 `analysis.logic`，而是取第一句完整语句作为短判断；若不存在结构化分析或第一句不可提取，则直接省略该行。这样可以保留个体化味道，同时避免把命格卡片撑得过长。

### D4. 缺失命格数据时不渲染命格解读区块

`ming_ge` 是区块存在的前提。若没有 `ming_ge`，打印布局保持现状，不插入空卡片或占位文案。这样对旧快照和异常数据更稳，也不会在 PDF 中制造空洞版面。

## Risks / Trade-offs

- [短句提取不够完美] → `analysis.logic` 的首句有时可能偏长或不够像“命格落点”。Mitigation：只把它作为可选补充，主信息仍以 `ming_ge` / `ming_ge_desc` 为主。
- [内容重复感] → 命格解读与命局总览都可能提到主格。Mitigation：命格卡片控制在短格式，避免重复整段分析。
- [旧数据缺字段] → 老结果若没有 `ming_ge` 将不显示区块。Mitigation：显式允许缺省，不影响原有 PDF 导出。
- [Prop 传递增加] → `ResultPage` 需要把命格数据下传给 `PrintLayout`。Mitigation：仅新增两个只读 props，影响范围有限。

## Migration Plan

无需迁移。该变更为纯前端渲染增强：
- 新代码上线后，新旧报告都可直接使用现有结果字段渲染
- 如需回滚，只需移除 `PrintLayout` 的命格区块与对应 props 传递
