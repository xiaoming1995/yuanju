# 资源化八字历史流引擎改动

## Requirements

1. 数据库解绑查重限制后，用户任何一次基于明确授权发起且意愿留存的“起盘动作”，将在数据底层以完全新生的 `chart_id` 快照形态保留；并如实呈列并存记录于前端展现。
2. 以历史快照或以实时起盘结果页呼出的任何后续 `AI 解读` 推理分析，皆严密下沉并针对该起单向指定的 `chart_id` 作为唯一判定其“此命盘是否已被读”和绑定附带成果回传的外键。不接受其余等同生辰带来的缓存漂移或穿透借用。
3. 支持从历史记录恢复并进入某张未经解读解析的图谱后，可顺理成章调用报告补打操作并挂载此节点本身完成；规避曾发无参导致 400 不接受等相关 Bug 错乱。

## Context & Tradeoffs
相较于曾设想的合并聚力打通思路，此举虽让用户个人历史留底可能产生“同日出生反复起堆叠了数百条毫无差异但独立且各自携带长卷宗”式的空间冗余情况——但考虑到我们现时身处纯文本轻量级别存储领域、更应最大程度维护用户分对象（即使同生辰也可能有为多人分次占卜之所需）的行为孤立独立感，因这种分离能为下步类似增加入“受测人姓名/称呼”标记系统等进阶功能直接铺设完备的前置空间。

## User Flow

- **Flow A - 无尽计算**：登录账户。多次反复为 `1992-05-18 男` 这个参数录入排盘 -> 记录历史将确切出现对应多少次同名记录快件。
- **Flow B - 分支生成**：在一览前项排盘的结果中按动一次“生成解读”，模型缓存后立刻渲染出文字。然后再点选另一条刚生成并回访历史页打开相同的八字卡片再“按动生成解读” -> 客户端将依然需要挂机等待全新的深推链产出并无涉上一条数据。

## Data Model

取消复合独立限制：
```sql
ALTER TABLE bazi_charts DROP CONSTRAINT IF EXISTS bazi_charts_chart_hash_user_id_key;
DROP INDEX IF EXISTS idx_bazi_charts_hash_user;
```
