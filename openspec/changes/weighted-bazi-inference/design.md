## Context

当前 `buildBaziPrompt()` 的 CoT 第一步是线性推断：先格局、再调候整合、再神煞点评。三个模块之间没有明确的权重关系，AI 自行决定最终喜忌神方向，导致结论倾向子平真诠格局派而非调候实战体系。

本变更仅修改 `report_service.go` 中的 Prompt 文本，不涉及算法层或 API 层，影响范围最小。

## Goals / Non-Goals

**Goals:**
- 在 CoT 第一步引入三模块独立评分 + 加权合并步骤
- 调候用神（65%）> 格局（25%）> 神煞（10%）固定权重
- 模块间矛盾时权重高者胜出，结论在报告中可见
- `analysis.logic` 包含加权推断说明，用户可理解推导过程

**Non-Goals:**
- 不实现动态权重（按月令调整）——本版本固定比例
- 不修改知识库内容（`kb_gejv`、`kb_tiaohou` 由管理员维护）
- 不改变 API 响应结构
- 不修改前端展示逻辑

## Decisions

### 决策1：权重表达方式——票数制 vs 百分比描述

**选择：票数制（量化打分）**

CoT 中要求 AI 对每个五行/天干输出"喜 +N / 忌 -N"的分值，最终合计后判断。

```
调候（65%）：喜丙火 → +65
格局（25%）：忌丙火 → -25  
神煞（10%）：中性   → 0
合计：丙火 = +40（喜，调候主导，格局有出入已注明）
```

**舍弃方案：** 只写"调候优先"文字描述——过于模糊，AI 执行一致性低。票数制让 AI 有明确的计算路径。

### 决策2：CoT 结构改写方式

**选择：保留现有第一步框架，在 a/b/c 内部扩充打分指令**

现有结构：
- a. 格局判断
- b. 调候整合  
- c. 神煞点评

新结构：
- a. 调候用神评分（65票）→ 输出每个五行的喜/忌票数
- b. 格局判断评分（25票）→ 输出每个五行的喜/忌票数
- c. 神煞综合评分（10票）→ 输出每个五行的喜/忌票数
- d. 加权合并 → 各五行票数求和，正数为喜、负数为忌、有矛盾标注

### 决策3：报告可见层的表达方式

在 `analysis.logic` 字段要求中新增：

> 「首段需包含加权推断说明：以调候（65%）为主导，格局（25%）次之，神煞（10%）辅助。最终用神为[X]，其中[模块]与[模块]有出入，以调候为准。」

## Risks / Trade-offs

- **Token 消耗增加** → 票数制 CoT 会增加约 200-400 tokens 的隐性计算，但在 `max_tokens=12000` 范围内无影响
- **AI 执行一致性** → 部分模型可能跳过票数计算直接给结论，建议在 `⚠️` 提示中强调不可跳过
- **矛盾标注质量** → 矛盾说明若过于技术性，用户难以理解；报告层面需要通俗化表达

## Migration Plan

1. 修改 `report_service.go` → `buildBaziPrompt()` 中第一步 CoT 文本
2. 修改第二步报告要求，在 `analysis.logic` 说明中添加加权推断段落要求
3. 编译验证
4. 重启后端服务
5. 手动触发一次报告生成，检查 `analysis.logic` 输出是否包含加权推断说明

回滚：还原 `report_service.go` 中对应文本块即可，无数据库变更。
