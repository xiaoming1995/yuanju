## Context

在华人文化中，许多人习惯使用农历（阴历）记录生辰。当前 `HomePage.tsx` 仅提供公历（阳历）的选择器，用户若只知农历，则必须借助第三方工具先转换，流程繁琐。

同时，后端使用的历法引擎 `github.com/6tail/lunar-go` 具备非常完备的阳历/农历互转能力，且准确支持复杂的农历「闰月」规则，我们只需在前后端接口上暴露并在计算前做一环预处理。

## Goals / Non-Goals

**Goals:**
- 前端生辰表单提供公历 / 农历无缝切换选项。
- 农历模式下，年份、月份下拉菜单能够联动计算出当年是否有「闰月」（例如 "闰四月"），并在下拉列表中正确渲染。
- 农历模式下，日期下拉菜单能结合当前农历年月大小正确限制（通常是大月30天、小月29天）。
- 后端能识别传入的数据是公历还是带有闰月标记的农历，若是农历，统一借助引擎在计算八字前转化为对应的公历再行推演。

**Non-Goals:**
- 不修改当前已持久化数据库表结构中的时间格式（数据库中统一存转换后的“真实客观时间”即对应的公历形式，方便查询与管理，但会缺失原始输入是农历的标记，当前业务下这可接受，算命结果是一致的）。
- 不把历史记录重新展示为农历（仅保证起盘那一刻输入便利，目前不做历史回看时的历法保持）。

## Decisions

### 前端状态与「年-月」联动计算
- **决策**: 引入 `lunar-javascript` (与后端的 `lunar-go` 同源) 库进行前端历期展示的推算。
- **Rationale**: 农历闰月的推算算法极其复杂（以地球绕日运行角度度量），自行手写或穷举极容易错。`lunar-javascript`(`lunar-typescript`) 是同一个作者的前端实现版。我们可以前端利用它直接算出“用户选的某一年有没有闰月”、“用户选的那个特殊农历月有29天还是30天”。
- **Alternative**: 前后端分离计算。前端 onChange 时调用后端一个 `/api/bazi/lunar-info?year=xxx` 的新接口查询当年农历详情。但网络延迟会导致下拉框闪烁、选项渲染慢。考虑到纯前端库体积可控并极大提升体验，选择前端直接引入相同算法推演。

### 后端 API 请求规范修改
- **决策**: 向 `POST /api/bazi/calculate` 新增如下参数：
  - `calendar_type`: `"solar" | "lunar"`（默认为 `solar` 以兼容旧版本）
  - `is_leap_month`: `bool`，标明此选择月份是否为闰月
- **Rationale**: 参数清晰分离，后续八字底层 `engine.go` 看到要是 `lunar`，可以直接通过 `calendar.NewLunar(year, isLeap? -month: month, day, hour, ...).GetSolar()` 拿到对应公历日期。

## Risks / Trade-offs

- [前端包体积增大] -> `lunar-javascript` 大约数 MB。若要优化可考虑动态 `import()` 或寻找其精简版。由于现在是管理端和单页重算命应用，当前不敏感。
- [真太阳时与农历公历节点偏差] -> 农历转为公历后再统一结合降生地的经度跑真太阳时校准，这符合经典天文历法的顺向推算逻辑，无额外风险。
