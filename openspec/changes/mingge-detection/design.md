## Context

缘聚命盘结果页目前已展示四柱、五行、用神忌神、神煞、调候用神等信息，但缺少命格（格局）这一核心定性结论。格局是传统命理中统领全局的关键，专业命理师在解盘时首要判断的就是格局归属。后端 BaziResult 结构体目前无任何格局相关字段，需从零实现。

## Goals / Non-Goals

**Goals:**
- 实现七优先级透干取格算法，在后端 `pkg/bazi/mingge.go` 中准确识别命格
- 将格名（ming_ge）和格局说明（ming_ge_desc）写入 BaziResult，随 calculate API 一并返回
- 前端顶部 Badge 展示格名，点击弹出 Modal 显示说明文字

**Non-Goals:**
- 不实现从强/从弱等从格（外格）——算法争议大，留作后续迭代
- 不单独为命格增加数据库字段，result_json JSONB 自动承接
- 不修改历史命盘的已存 result_json（旧命盘无此字段则前端降级不展示）

## Decisions

### 决策 1：透干取格的格神判定使用七优先级算法

**选择**：按以下优先级依次检查，取第一个命中的结果：
1. 月支主气藏干（司令）透出到月干（月柱自透）
2. 月支任意藏干透出到年/日/时干（月支透它干）
3. 其它三柱地支主气藏干透出到月干（它支透月干）
4. 其它三柱地支任意藏干透出到任意天干（它支透它干）
5. 多透时以透出次数最多、五行力量最强者为格神
6. 无透格神时，检查四柱是否构成三合局/三会局（取合局所成五行对应十神为格神）
7. 兜底：统计干支中各十神出现总数，满四个以上者取频率最高者成格

**理由**：此七优先级体系为当代主流命理实践所采用，规则清晰可编码，覆盖绝大多数命局，与知名命理软件（如大六壬）的取格逻辑基本对齐。

**替代方案**：仅用月令主气一刀切（最简单），但无法正确处理月支藏干不透出、气势转移等情况，误判率高。

### 决策 2：格名采用十神 + "格" 后缀的命名规则

格神确定后，将其相对日主的十神关系加"格"字作为格名：
- 正官 → 正官格
- 七杀 → 七杀格
- 食神 → 食神格
- 伤官 → 伤官格
- 正财 → 正财格
- 偏财 → 偏财格
- 正印 → 正印格
- 偏印 → 偏印格
- 比肩 → 建禄格（月令比肩特称）
- 劫财 → 月刃格（月令劫财特称）

**理由**：符合传统命理命名惯例，用户认知成本最低。建禄/月刃是专业术语，须做特殊映射。

### 决策 3：格局说明文字内置于代码字典，不入库

每个格名对应一段固定的说明文字，以 Go 的 `map[string]string` 形式硬编码在 `mingge.go` 中，随 API 响应直接返回。

**理由**：格局说明文字语义稳定，无需动态编辑，内置代码最简单、最快，避免额外数据库查询。若未来需要运营动态编辑，再迁移到 Admin 配置表。

### 决策 4：前端复用神煞注解 Modal 组件

格名 Badge 点击后弹出的说明 Modal 直接复用 `shensha-modal-*` 的 CSS 类和弹出逻辑，不新建组件。

**理由**：两者信息结构一致（名称 + 说明文字），复用可零成本完成 UI，保持设计系统一致性。

## Risks / Trade-offs

- **算法争议**：命理取格存在流派差异，七优先级方案会有小概率与某些流派判断不同 → 降低风险：在说明文字中注明"格局判定基于月令透干优先法"
- **旧命盘无格局字段**：历史 result_json 无 ming_ge 字段，前端需做降级处理（格名 Badge 不展示）→ 已在前端逻辑中用可选链处理
- **边界命局**：极少数命局七个优先级均无法确定格局（如四柱无任何透干且无合局）→ 返回 `ming_ge: "杂气格"` 作为兜底标签
