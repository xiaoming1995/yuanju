## Context

当前 AI 命理报告由 `report_service.go` 中的 `buildBaziPrompt()` 构建 Prompt，调用 `callAI()` 生成结构化 JSON（含 `analysis` + 六个 `chapters`）。引擎计算的调候用神（《穷通宝鉴》）和月令格局（《子平真诠》）均未被利用，Prompt 指令存在风格冲突和章节泛化问题。

## Goals / Non-Goals

**Goals:**
- 在后端引擎新增调候查表（`tiaohou.go`），精确输出日主 × 月令 → 调候用神
- 重写 `buildBaziPrompt()`：统一现代解读风格、注入当前年份、加入章节数据锚点和格局推断 CoT 步骤
- 调整调用参数：MaxTokens 6000、Temperature 0.75
- `BaziResult` 新增 `Tiaohou` 字段，随 calculate API 一并返回

**Non-Goals:**
- 不修改前端报告渲染逻辑
- 不修改数据库 Schema 或 API 路由
- 不实现后端格局引擎精算（格局由 AI 推算）
- 不修改用神缓存逻辑

## Decisions

### 1. 调候引擎：新建 `tiaohou.go` 查表文件

**决策**：新建独立文件 `backend/pkg/bazi/tiaohou.go`，实现 `LookupTiaohou(dayGan, monthZhi string) string` 函数。

查表结构：`map[string]map[string]string`，外层键为日主天干（10个），内层键为月支（12个），值为调候用神说明文本（如：`"癸水（首选）、丙火（次选）"`）。

`engine.go` 中在 `Calculate()` 末尾调用 `LookupTiaohou()` 并填入 `BaziResult.Tiaohou` 字段。

**为何选择硬编码查表而非算法推导**：调候是《穷通宝鉴》的经验总结，120条规则每条均有细微差别，不适合用算法概括；查表是最准确且可维护的实现方式。

### 2. 月令格局：AI 推断（非引擎硬算）

**决策**：格局判断逻辑通过 Prompt CoT 指令实现，不在后端添加格局查表。

**理由**：月令主气十神已由引擎精算注入 Prompt，AI 基于此推断格局（正官格/七杀格等）出错概率低；且格局判断需要结合天干透出情况（成格/破格），纯查表无法处理透干条件，反而引入新的复杂度。

### 3. Prompt 重写策略：System / User 消息分离

**当前问题**：System message 说「通俗易懂」，User prompt 又要求「先写专业术语」，AI 在两种风格间摇摆。

**新策略**：
- **System message**：定义身份（专业命理师）+ 风格锚（现代解读风格，通俗直接，结论先行，术语作精准点缀）+ 输出格式要求（必须输出合法JSON）
- **User message**：只含命盘数据 + CoT 步骤指令，移除所有风格定义内容

### 4. 新增 CoT 步骤：格局 + 调候整合判断

在原有「第一步：综合精算数据」基础上，新增两项明确子任务：

```
a. 月令格局推断：基于月支藏干主气[X]=[十神]，判断格局名称（如：月支亥水藏壬，壬对甲木=正印，此命为正印格）
b. 调候整合：参考引擎调候用神[X]，结合格局用神，综合确定最终喜用神
```

此步骤在心中完成，结论融入 `analysis.logic`。

### 5. 章节数据锚点设计

每章指令新增「重点参考」提示，将分析锚定于具体精算字段：

| 章节 | 核心锚点 |
|------|---------|
| 性格特质 | 日主五行特质、日支星运、比劫/印绶十神力量 |
| 感情运势 | 男命财星/女命官杀的位置与力量、日支、桃花/红鸾神煞 |
| 事业财运 | 官杀/食伤的透干情况、财星旺衰、文昌/天乙贵人 |
| 健康提示 | 五行过旺/过衰的脏腑对应、旬空地支 |
| 大运走势 | 当前年份([YEAR])推算当前大运步、近2步大运十神与日主关系 |
| 命理分身 | 日主+格局+神煞特质与名人参考库匹配 |

### 6. 参数调整

| 参数 | 旧值 | 新值 | 原因 |
|------|------|------|------|
| MaxTokens | 4500 | 6000 | 6章×(100+350)字 + 调候格局内容，原上限过紧 |
| Temperature | 1.0 | 0.75 | 命理报告需要稳定性，降低逻辑跳脱风险 |

## Risks / Trade-offs

- **调候表录入工作量**：120条规则需手工逐一核对《穷通宝鉴》原文，若录错会产生错误的调候用神 → 可在 `engine_test.go` 中添加典型案例测试覆盖
- **Prompt 变长增加 Token 消耗**：调候注入 + 章节锚点约增加 200~300 prompt token → MaxTokens 同步提升到 6000 可覆盖，总消耗可控
- **格局推断 AI 可能出错**：边缘命局（如从强格/从弱格/化气格）AI 判断可能偏差 → 这类特殊格局在 Prompt 中作为补充说明提醒 AI 注意，并在未来版本引入引擎校验
- **缓存报告不受益**：已生成的旧报告命中缓存，不会重新生成 → 属于预期行为，新报告质量提升对新用户立即生效
