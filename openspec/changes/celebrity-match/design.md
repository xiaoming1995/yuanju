## Context

当前 Yuanju 的 AI 报告在排盘核心结果出来后，会进行一次单次对话调用，按照写死的 Prompt 模板让 AI 拆分为四个章节解读命盘（性格、感情、事业、健康）。为了增加代入感和社交趣味，我们提出引入名人相似度匹配。由于大模型存在知识幻觉，无法准确获知所有名人的确切四柱参数，因此必须依赖平台内部构建的高质量名人库，并在发送给大模型的前置上下文中一并提供。

## Goals / Non-Goals

**Goals:**
1. 设计稳固的 `celebrities` (名人表) 数据库结构。
2. 实现后端接口及 Admin 前端管理页面，以支持手工录入和维护名人库数据。
3. 改造现有 `bazi/report` 生成流程：读取全体 active 的名人信息，拼接在与大模型的对话 Prompt 中，要求其从中选取一位并追加在第五章输出。

**Non-Goals:**
1. 不涉及前端普通用户侧的大型改版（依赖目前的 Markdown 渲染逻辑即可）。
2. 不要求算法层面严格计算分数，全部交由大模型的推理能力。
3. 初期不实现在线获取或爬虫采集名人八字的功能，完全由人工 Admin 添加。

## Decisions

1. **数据库表设计**: 
   表名 `celebrity_records`。包含：
   - `id` (UUID)
   - `name` (String: 姓名)
   - `gender` (Enum: 男/女)
   - `traits` (String/JSONB: 核心八字特征或标签，如“丁火生于酉月，财多身弱”等)
   - `career` (String: 职业或代表性身份)
   - `active` (Boolean: 是否在匹配池中启用)

   *讨论点：由于依赖大模型推理，暂不严格计算完整四柱天干地支，主要存储 `traits` 文本描述。*

2. **请求组装机制**: 
   - 每次生成 `bazi_charts` 的 AI 报告前，执行 `Select name, traits, career FROM celebrity_records WHERE active = true` 取得全量活跃名人（预计约 50-100 行，轻量级）。
   - 将这批数据序列化拼接进 `reportPrompt` 中：“请基于该命盘，从以下名人列表中选择出与命主相似度最高的一位，在此报告的最后输出第五章《命理分身》。候选人列表：\n...”

3. **Admin 后台**: 
   复用当前的 `AdminLLMPage` 风格组件，新增 `AdminCelebrityPage` 并配置路由。包括列表展示（分页可选，暂定全量）、新增弹窗、编辑/启用状态切换。

4. **预置数据生成机制（AI 自动收集）**:
   为了更轻松地填充知识库，将在 Admin 后台增加 `/api/admin/celebrities/ai-generate` 接口。管理员可输入主题边界与数量，利用系统已激活的大模型生成对应的结构化名人数据并批量自动落库。

## Risks / Trade-offs

1. **上下文长度 (Token) 消耗**：注入 100 位名人的文本可能增加每次生成的 Prompt 长度约 500-1000 Token。考虑到当前使用 DeepSeek 等大模型的成本极低，且具有较好的长文本处理能力，这笔开支完全可接受。
2. **大模型不遵循指令风险**：AI 可能未能将名人输出为“第五章”，导致 MD 解析或者阅读感违和。
   - 缓解策略：严格限定系统 Prompt 的输出 JSON 格式或 Markdown 章节层级，并给出 Few-shot 示例。
