## Context

当前算法层已通过 `DetectMingGe()` 计算 `MingGe` / `MingGeDesc`，见 `backend/pkg/bazi/engine.go`；但原局 AI 报告链路 `buildBaziPrompt()` 尚未把这两个字段注入 Prompt，反而继续要求 LLM 在格局模块中按 `kb_gejv` 公式重新定格。结果是：

```text
算法层：MingGe（顶部 Badge 展示）
    ↓
AI 报告 Prompt：未使用 MingGe
    ↓
LLM 依据 kb_gejv 再定一次格
    ↓
analysis.logic 可能与 MingGe 冲突
```

本变更目标不是弱化格局，而是把职责拆清：

- 算法层负责判定主格名称
- LLM 负责解释主格是否成格、是否受破、与调候/喜用神的关系如何

## Goals / Non-Goals

**Goals:**
- 让 `MingGe` 成为 AI 报告中的唯一主格真源
- 让 `analysis.logic` 首段稳定围绕主格展开
- 避免 `kb_gejv` 与主 Prompt 出现“一个让 AI 以系统主格为准，另一个仍教 AI 重新定格”的冲突
- 保持现有响应结构、前端展示结构和解析链路稳定

**Non-Goals:**
- 不修改 `DetectMingGe()` 算法本身
- 不新增前端主格展示组件
- 不重构 `content_structured` 结构
- 不强制重写历史已生成报告

## Decisions

### 决策 1：`MingGe` 作为 AI 报告中的唯一主格真源

**选择**：在 `buildBaziPrompt()` 中新增 `[系统定格结果]` 区块，显式注入：

- `MingGe`
- `MingGeDesc`
- “以下解读必须以此主格为准，不得重新改判格名”的硬约束

**理由**：这能最小代价打通现有算法层与 AI 报告层，消除“双轨格局判断”。

### 决策 2：格局模块从“判格”改为“释格”

**选择**：保留 Prompt 中的“格局模块”，但职责改为：

- 说明该主格在原局中是否成格 / 破格 / 有救 / 偏弱
- 解释月令、透干、生扶、制化为何支持或削弱这个主格
- 若存在其它结构，只允许写成“兼带某某倾向”，不得覆盖主格

**理由**：LLM 仍保留命理解释力，但不再拥有格局命名权。

### 决策 3：`analysis.logic` 固定“主格优先”顺序

**选择**：要求 `analysis.logic` 首段按固定顺序展开：

1. 先写“此命以【X格】立局”
2. 再写该格局在原局中属于成格 / 破格 / 有救 / 偏弱中的哪一种
3. 再写该格局与调候、最终喜用神之间的关系

**理由**：固定顺序能显著提高输出稳定性，避免模型绕开主格直接讲喜忌。

### 决策 4：`kb_gejv` 改为解释型知识库，不再承担重新定格职责

**选择**：将 `kb_gejv` 从“如何按月令透干定格”改为“系统主格如何解释”，内容聚焦：

- 成格 / 破格 / 有救 / 偏弱的判断口径
- 各主格的解读重点
- 格局与调候、喜用神、大运的关系
- 明确禁止覆盖系统主格

**理由**：若不调整知识库，主 Prompt 和 `kb_gejv` 会继续对模型发出相反指令。

### 决策 5：历史旧快照无 `MingGe` 时保留降级路径

**选择**：当旧 `result_json` 中无 `MingGe` 时，Prompt 允许回退到旧格局推理路径；对新快照则严格启用“系统主格为准”。

**理由**：不强制迁移历史命盘，降低本轮变更风险。后续可独立决定是否批量补算旧快照。

## Risks / Trade-offs

- **旧报告与新报告口径差异**：历史已生成报告仍可能沿用旧逻辑 → 接受现状，本轮不主动重建历史报告
- **知识库升级影响运营自定义内容**：若管理员已手工修改 `kb_gejv`，自动覆盖会破坏现网策略 → 默认只更新 seed/默认值，不强制覆盖自定义内容
- **旧快照降级路径仍可能双轨**：少量老命盘若无 `MingGe` 仍可能走旧逻辑 → 通过降级兼容保证稳定，后续再决定是否补算
- **输出更收敛、文风自由度下降**：LLM 不再自由定格，表达空间会缩小 → 换来口径统一和专业可信度提升，值得

## Migration Plan

1. 修改 `buildBaziPrompt()`，注入 `[系统定格结果]` 区块
2. 将第一步中的格局模块改写为“解释主格”而非“重新定格”
3. 将第二步 `analysis.logic` 改写为“主格优先”的固定展开顺序
4. 更新默认 `kb_gejv` 内容为解释型短版；若存在 Admin 自定义内容，则保留现有值并由运营自行确认是否替换
5. 生成若干新报告样本，人工验证：
   - 顶部命格 Badge 与 `analysis.logic` 主格一致
   - 出现兼象时以“兼带某某倾向”表达，而非覆盖主格
   - 调候与格局冲突时，文案明确主次

回滚：恢复 `buildBaziPrompt()` 旧版格局指令，并恢复旧版 `kb_gejv` 默认内容即可；无 schema 回滚需求。
