## Context

`past-year-events` 上线后命主反馈推算不准。代码体检发现 `pkg/bazi/event_signals.go`(458 行) 的规则集只覆盖了不到一半的常用命理参数：神煞引擎完全未接入、用神/忌神比对缺失、流年只与日柱+大运互动（年/月/时柱不扫）、空亡/三会/三刑/伏吟/反吟/大运合化全部缺失，且 `dayMasterStrength` 是粗粒度三档判定。AI Prompt 拿到的信号矩阵不完整，自然写不准。

本次改造的目标：在不破坏既有 API/缓存契约的前提下，把 `event_signals.go` 升级为覆盖十大命理维度的完整信号引擎，并把对应的上下文（格局/用忌神/合化/身强弱明细）注入 Prompt，让 AI 写出准确性显著提升的批断。

## Goals / Non-Goals

**Goals:**
- 算法层捕获率提升：真实大事年的算法信号产出比例显著上升（回归测试集量化）
- 吉凶判断方向正确：通过 polarity 字段把"流年五行 vs 用神/忌神"作为每个信号的基底色，避免 AI 望文生义
- 大运整体定调可解释：把大运合化、神煞、用神匹配作为大运 summary 的输入
- 已有神煞引擎（`shensha.go` / `shensha_dayun.go`）零改动接入
- 不破坏既有 API：`POST /api/bazi/past-events-stream/:chart_id` 路径与请求/响应顶层结构不变；`signals` 数组项形态向后兼容（新字段 optional）

**Non-Goals:**
- 流月/流日级别的事件细化（保持原 spec non-goal）
- 用户输入已知事件做闭环验证（仍为纯推算）
- 重写八字格局判定算法（用现有 result 字段中已有的格局信息；若无则跳过格局描述）
- 替换 AI 模型或调整 SSE 协议
- 自动失效既有缓存（保留 Admin 手动清缓存入口）

## Decisions

### 决策 1：EventSignal 字段扩展，不破坏向后兼容

**选择**：在 `EventSignal` 上增加两个 optional 字段：
```go
type EventSignal struct {
    Type     string `json:"type"`
    Evidence string `json:"evidence"`
    Polarity string `json:"polarity,omitempty"` // "吉" / "凶" / "中性"
    Source   string `json:"source,omitempty"`   // "神煞" / "柱位互动" / "合化" / "空亡" / "刑" / "会" / "伏吟" / "用神基底"
}
```

**放弃**：另起一个 `EventSignalV2` 结构 + 新 API 端点。

**原因**：现有 SSE 输出结构是 AI JSON，前端展示只读 `type/narrative`。`signals` 字段是数组，新增 optional 键不会破坏前端解析；polarity/source 仅在算法→Prompt 阶段使用，不暴露给前端。

---

### 决策 2：用神/忌神基底色实现路径

**选择**：在 `GetYearEventSignals` 入口处先计算"流年五行 vs `natal.Yongshen` / `natal.Jishen`"的吉凶倾向（`yongshenScore`），然后：
1. 输出一条独立 `Type=用神基底` 信号，作为 AI Prompt 的整年定调
2. 后续所有规则信号都根据 `yongshenScore` 与信号本身性质共同决定 `Polarity`
   - 例：财星透干（信号本身偏吉）+ 财为忌神（基底偏凶）→ Polarity="凶"，Evidence 注明"财星虽透但为忌神，财来财去/破耗"

**放弃**：仅给 AI 文字 hint（"流年透忌神"），让 AI 自己推 polarity。

**原因**：实测发现 DeepSeek 在多信号聚合时容易顾此失彼，强制算法层打 polarity 标签可压缩幻觉空间。

---

### 决策 3：神煞接入策略——复用 `shensha_dayun.go`，不重写

**选择**：调用现有 `bazi.CalcDayunShensha(...)` 得到大运/流年级别的神煞列表，再按白名单（天乙贵人/羊刃/华盖/白虎/丧门/吊客/天月德/红艳/将星/勾绞/孤辰寡宿）转换为 `EventSignal`，每条神煞带预设 `Polarity`：

| 神煞 | Polarity 默认 | 类型映射 |
|------|--------------|---------|
| 天乙贵人/天德/月德 | 吉 | 事业 |
| 羊刃/血刃 | 凶 | 健康 |
| 白虎/丧门/吊客 | 凶 | 健康/迁变 |
| 华盖 | 中性 | 迁变 |
| 红艳 | 中性 | 婚恋 |
| 将星 | 吉 | 事业 |
| 勾绞 | 凶 | 综合变动 |

**放弃**：在 event_signals.go 重新实现神煞规则。

**原因**：`shensha.go` + `shensha_dayun.go` 共 43K 行已实现并经过 admin 配置打磨，重复实现违反 DRY。

---

### 决策 4：柱位互动检测的范围

**选择**：扫描流年地支与所有四柱地支（年/月/日/时）+ 大运地支的 5 种关系：
- 六合、六冲、相刑、伏吟（地支相同）、反吟（天克地冲全柱）

按"哪一柱被扫中"打事件类型映射：
- 年柱 → 祖荫/根基/家族
- 月柱 → 行业/工作/父母
- 日柱 → 自身/婚恋（保留现有逻辑）
- 时柱 → 子女/晚景

**放弃**：扩展到藏干层级互动（如流年地支与日支藏干暗合）。

**原因**：藏干互动加权复杂、流派分歧大，先把主气层做扎实；后续若需要再迭代。

---

### 决策 5：大运合化判定的边界

**选择**：仅判"合而成局"的强合化（满足 3 条件）：
1. 大运天干与日干形成天干五合（甲己/乙庚/丙辛/丁壬/戊癸）
2. 月支或大运地支提供化神五行根气
3. 原局无强力反克（如丁壬合木，月支不能透出庚辛）

满足全部 → 输出 `Type=大运合化` 信号，标注转化五行；不满足 → 仅作为 `DayunHuahe` 描述提示 AI"日干被合住但未化"。

**放弃**：使用经典"逢龙则化"等更严格的判定。

**原因**：宽严平衡——纯五合不化太松，逢龙则化太严，月支化神根 + 无反克是大多数命理流派认可的中庸标准。

---

### 决策 6：身强弱重写为加权评分

**选择**：保留 `dayMasterStrength` 函数签名，内部重写：
```
得令权重 ×5（月支与日干同气/生我/比我/泄我/克我）
得地权重 ×3（其余地支本气与日干关系）
得势权重 ×2（藏干透出 + 天干生扶/克泄）
```
评分 → 5 档：极强(≥10)/强(5-9)/中和(-4-4)/弱(-9--5)/极弱(≤-10)。
返回值由原 `"strong"|"weak"|"neutral"` 扩展为 `"vstrong"|"strong"|"neutral"|"weak"|"vweak"`。

**放弃**：保留三档。

**原因**：现有"边界 strong/weak"误判会连带把财官印的吉凶推倒；五档可拉开梯度，让"中和"档真正稳定。

---

### 决策 7：合冲并见的优先级与对消逻辑

**选择**：扩展现有"婚恋_冲 + 婚恋_合 → 婚恋_变"模式，覆盖三组合冲并见情形：
- 大运地支合住日支 + 流年地支冲日支 → "感情合冲交织"信号（保留两条 evidence）
- 流年地支同时引动两个三合/三会局 → 输出 `Type=综合变动`，evidence 列出双局信息
- 流年与某柱伏吟 + 同时有冲该柱 → 优先输出"伏吟+冲"组合信号（凶）

**放弃**：完全机械的"合 + 冲 = 抵消"。

**原因**：合冲并见在命理上多解释为"事件张力极大、方向不定"，不应抵消，而应当被 AI 重点叙述。

---

### 决策 8：Prompt 升级与旧缓存

**选择**：
- 升级 `defaultPastEventsPrompt`，新增模板变量 `{{.YongshenInfo}}`、`{{.GejuSummary}}`、`{{.DayunHuahe}}`、`{{.StrengthDetail}}`
- 用现有"非 dayun_summaries 即升级"的 SQL 模式（参考 `database.go:238`）：
  ```sql
  UPDATE ai_prompts SET content = $1
  WHERE module = 'past_events'
    AND content NOT LIKE '%YongshenInfo%'
  ```
- 已生成的 `ai_past_events` 缓存不主动失效（避免命主重新等待 30 秒），由 Admin 清缓存按钮触发

**放弃**：自动清空所有 `ai_past_events` 表数据。

**原因**：避免一次部署造成所有命主报告全部需要重新 AI 生成，对成本与体验都不好；旧报告"不够准但可读"，新报告"更准"，并行存在过渡期可接受。

---

### 决策 9：回归测试集

**选择**：在 `event_signals_test.go` 中维护 5–10 个有公共已知大事的真实命例（公众人物或命主授权的脱敏样本），断言：
1. 大事年信号数量 > 0
2. 大事年 Polarity 与实际事件吉凶方向一致
3. 历史平年信号数量适中（避免过拟合）

**放弃**：纯随机生成命例。

**原因**：命理算法的"准"无法用纯单元测试覆盖；以"有真实事件锚点的命例"作回归基线，是工程化可接受的折中。

## Risks / Trade-offs

- **新字段被前端旧版本忽略** → 已通过 optional JSON 字段保证向后兼容；前端解析 narrative 即可，polarity/source 留作未来 UI 扩展
- **Prompt 升级触发缓存"新旧报告不一致"** → 接受过渡期；Admin 提供清缓存按钮主动重置
- **神煞接入引入大量信号噪声** → 用白名单（约 10 种）控制；其他神煞仅作为 evidence 补充而不单独成 signal
- **加权身强弱评分门槛主观** → 阈值（10/5/-5/-10）通过 `algo_config` 暴露给 Admin 可调
- **大运合化判定派系分歧** → 在决策 5 中已采取中庸路线；后续如需切换严格规则，可通过 `algo_config` 加开关
- **回归测试样本规模小** → 首版 5-10 例先建立基线；后续按命主反馈持续扩充
- **AI 在 polarity 字段冲突时混乱**（基底凶但单条信号吉）→ 在 Prompt 中显式约束："基底色优先，单条信号 polarity 是局部判断；冲突时以基底为主、注明转折"

## Migration Plan

1. **代码上线**：合并改造代码，部署后端服务
2. **DB 升级 SQL 自动执行**：`InitDB` 启动时按"不含 YongshenInfo 即升级"覆盖 prompt
3. **观察期**：保留旧 `ai_past_events` 缓存，新生成的报告将使用新 prompt + 新信号
4. **Admin 通知**：通过 admin 页面提示"过往事件推算算法已升级，可选择批量清缓存让所有命主重新生成报告"，不强制触发
5. **回滚策略**：若发现严重问题，revert 代码 + 通过 admin 页面手动恢复旧 prompt 内容
