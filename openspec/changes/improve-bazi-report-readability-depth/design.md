## Context

命理解读报告目前已经支持 `content_structured`，前端可在精简和专业两种模式之间切换。实际体验中，报告默认进入精简模式，用户第一眼只看到短结论；后端 Prompt 对“命局分析总览”有 `300-500字` 限制，但对各专业章节没有稳定字数和段落结构约束，导致详解内容容易偏短。前端报告正文固定为 `15px`，在长篇中文阅读场景下显得密集。

本次变更不重做报告结构，只在现有结构化报告之上提升“默认可读性”和“生成深度”。

## Goals / Non-Goals

**Goals:**

- 让新报告默认呈现更完整的专业详解，而不是默认短版。
- 提升报告正文阅读舒适度，尤其是中文长段落的字号、行高和层次。
- 让专业版章节稳定输出较完整内容，避免模型只给泛泛两三句话。
- 让术语解释更友好，专业判断后跟随普通用户能理解的现实含义。
- 保持旧报告、旧 `content` 降级渲染和现有 API schema 兼容。

**Non-Goals:**

- 不新增数据库字段。
- 不改变 `content_structured` 的 JSON 字段结构。
- 不引入新的前端 UI 组件库。
- 不改八字算法、喜忌判断、格局判断和大运算法本身。
- 不做历史报告批量重生成。

## Decisions

### D1：默认模式从精简改为专业详解

`ResultPage.tsx` 继续保留 `brief/detail` 双模式，但新格式报告默认使用 `detail`。用户仍可切回精简模式。

**理由**：当前用户反馈的“不够详情”首先来自默认只看见 `brief`。直接默认显示 `detail` 能最大程度复用已有结构，风险低，也不会影响旧报告降级路径。

**替代方案**：新增第三种“完整解读”模式，同时展示摘要和专业版。该方案信息更全，但会扩大 UI 和状态逻辑，本轮先不引入。

### D2：Prompt 明确专业章节字数和组织结构

在 `buildBaziPrompt()` 的章节指引中加入硬性输出规则：

- 精简版：每章约 `80-120字`，结论先行。
- 专业版：每章约 `220-350字`。
- 专业版必须包含“结论、命理依据、现实表现、建议”四类信息，但不要求输出机械小标题。
- 术语出现后必须用一句白话解释，例如“印星有力”后说明对应学习、证书、长辈资源或保护性资源。

**理由**：只提高 `MaxTokens` 不能保证内容变详细，模型需要明确的篇幅和内容结构约束。

**替代方案**：把章节拆成更多结构化字段，如 `basis`、`plainText`、`advice`。该方案更利于渲染，但会改变 API 结构，不适合本次小步优化。

### D3：命局分析总览放宽长度但保持克制

将 `analysis.logic` 从 `300-500字` 调整为约 `500-800字`，并要求先给判断，再讲依据，最后落到现实含义和建议。

**理由**：命局总览承担“为什么这么断”的解释功能，300 字以内容易只剩术语堆叠。800 字以内可以保证深度，同时避免报告过长。

### D4：前端排版只做局部增强

报告正文样式在 `ResultPage.css` 中局部调整，例如正文提升到 `16px`，行高提升到约 `1.9`，摘要、正文和模式切换文案保持一致的可读层级。移动端使用同一字号或轻微下调，但不低于当前可读性。

**理由**：报告页已有完整视觉体系，本次只处理阅读密度，不做大面积视觉重构。

## Risks / Trade-offs

- **Token 消耗增加** → 专业版字数增加会提高输出 token；通过保留精简版、限制专业版上限和不扩展 JSON schema 控制成本。
- **模型偶发不遵守字数** → Prompt 增加明确格式和字数要求，同时用测试覆盖 Prompt 文案关键约束。
- **默认专业版页面更长** → 用户可切回精简模式；默认详解符合“命理解读要详细”的主要诉求。
- **旧报告看起来变化有限** → 旧报告没有结构化详解内容时仍走降级渲染；本轮不重生成历史报告。

## Migration Plan

1. 更新后端 Prompt，保持返回结构不变。
2. 更新前端默认模式和报告正文样式。
3. 运行后端测试和前端构建。
4. 用一组固定生日生成新报告，验证默认进入专业详解、每章内容更完整、术语有白话解释。
5. 验证历史旧报告仍能降级展示。
