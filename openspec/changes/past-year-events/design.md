## Context

现有系统已支持「流年精批」（`POST /api/bazi/report-stream/:chart_id`），用户单次选取一个年份，AI 生成散文式运势报告并缓存于 `ai_liunian_reports`。该功能面向未来运势分析，无法满足用户"一眼纵览过往人生命理轨迹"的需求。

本变更新增独立功能模块：先由算法扫描所有过往流年，检测每年激活了哪些事件信号，再将结构化信号批量送入 AI，生成覆盖全部过往年份的时间轴式叙述报告。

## Goals / Non-Goals

**Goals:**
- 算法精确检测每一流年的事件信号（婚恋/事业/财运/健康/迁变）及其命理证据
- 单次 AI 调用批量生成所有过往年份（约 12岁起至当前年），每年 2-3 句叙述
- SSE 流式推送，避免长时间白屏等待
- 生成结果按命盘缓存，再次访问直接读取
- Admin 可管理 `past_events` Prompt 模板

**Non-Goals:**
- 未来年份预测（本功能仅覆盖过往年份，含当年）
- 用户输入已知事件进行验证（纯推算模式）
- 替代现有「流年精批」（两者独立共存）
- 流月级别的事件细化

## Decisions

### 决策 1：算法检测 + AI 组织语言（混合方案）

**选择**：`pkg/bazi/event_signals.go` 按规则计算信号，再将结构化信号列表注入 Prompt 供 AI 撰写。

**放弃**：纯 AI 生成（无算法预筛）。

**原因**：纯 AI 会产生幻觉（无中生有的命理证据）；算法预筛确保每一句叙述有命理依据，AI 只负责将「偏财透干 + 夫妻宫冲」翻译为自然语言，准确性和成本均优于纯 AI。

---

### 决策 2：单次 AI 调用覆盖所有年份

**选择**：构建一个包含所有过往年份信号的 Prompt，一次 AI 调用返回完整 JSON。

**放弃**：逐年调用 AI（每年一次请求）。

**原因**：逐年调用会产生 N × API 延迟（30-40 次请求），成本与等待时间不可接受；单次调用 output token 约 3000-5000，DeepSeek 完全可承受。

---

### 决策 3：最小年龄阈值 = 虚岁 12 岁

**原因**：12岁以前的命理事件对用户意义有限（以家庭/双亲为主），且推算准确性低。从 12 岁起算可覆盖青少年期之后有意义的人生阶段。具体开始年份由 `bazi.Calculate` 返回的 `LiuNianItem.Age >= 12` 过滤。

---

### 决策 4：缓存粒度 = 命盘级（chart_id），不分年份

**原因**：过往年份是固定的历史集合（不会新增），无需像流年精批那样逐年缓存。整个过往事件报告以一条记录存入 `ai_past_events` 表，`content_structured` 字段存 JSONB（年份数组）。用户重新排盘（新 chart_id）时自动重新生成。

---

### 决策 5：SSE 流式推送

**选择**：`/api/bazi/past-events-stream/:chart_id` 使用与 `report-stream` 相同的 SSE 架构。

**原因**：AI 单次输出 3000-5000 token，耗时约 15-30 秒，必须流式推送以避免用户长时间面对空屏。

---

### 决策 6：信号检测规则（事件类型 → 命理触发条件）

| 事件类型 | 触发条件 |
|---------|---------|
| 婚恋（男） | 偏财/正财透干；财星入日支；流年地支与日支六合/冲；桃花临命 |
| 婚恋（女） | 正官/七杀透干；官星入日支；流年地支与日支六合/冲；桃花临命 |
| 事业 | 官星/印星透干且为用神；驿马动；官印相生 |
| 财运 | 财星透干得令；财星被冲克（财损）；劫财夺财 |
| 健康 | 日主天干被克（流年天干克日干）；地支冲日支；病符/白虎/五鬼临命 |
| 迁变 | 驿马临命；华盖引动；流年与年柱相冲 |

## Risks / Trade-offs

- **Prompt 超长（高龄命主）** → 缓解：目前 DeepSeek context 128K，60年×约40 token/年=2400 token，安全边际充足；若将来需要限制，可截取近 40 年。
- **AI 质量差异** → 缓解：算法提供结构化证据，AI 只做语言组织，幻觉空间被压缩；Prompt 严格要求 JSON 格式输出，便于前端解析和二次校验。
- **信号规则不完整** → 缓解：首版仅覆盖五大事件类型，算法设计为可插拔扩展，后续可通过 Admin 调整；不触发任何信号的年份 AI 仍输出"该年较为平稳"的叙述。
- **缓存失效问题** → 缓解：Admin 页面提供手动清除缓存入口（复用现有 `AdminClearReportByChart` 模式）。
